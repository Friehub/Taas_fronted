# Running a Node

This guide takes you from zero to a running TaaS Gateway node. The target is a working Sovereign Node — a single-operator deployment. Enabling Mesh Network mode (P2P + TSS) is covered at the end once you have a running baseline.

---

## Prerequisites

| Tool | Minimum Version | Install |
| :--- | :--- | :--- |
| Rust | stable | `curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs \| sh` |
| Node.js | 22+ | [nodejs.org](https://nodejs.org) or `nvm install 22` |
| pnpm | 9+ | `npm install -g pnpm@9` |
| Docker | any | Optional. Required only for Redis-backed distributed state and integration tests. |

---

## Step 1 — Clone and Bootstrap

```bash
git clone https://github.com/Friehub/Taas.git
cd taas-gateway
./bootstrap.sh
```

`bootstrap.sh` installs all Node.js dependencies, builds the TypeScript sidecar, compiles the Rust workspace, and configures Husky pre-commit hooks. You do not need to run anything else after this completes.

---

## Step 2 — Initialize the Node Identity

The gateway stores all secrets in an encrypted Vault. Initialize it before first start:

```bash
./target/debug/hot-core init
```

The `init` command is interactive. It:

1. Prompts for a **Master Password** — this is the encryption key for the Vault. Choose a strong password and store it securely. If you lose it, all encrypted keys are unrecoverable.
2. Generates an **Ethereum identity** (secp256k1 private key). This becomes your node's `gatewayIdentity` address on-chain.
3. Generates a **BLS12-381 signing key** for TSS threshold signatures.
4. Generates a **secure admin token** for the Admin API.
5. Writes a `config.toml` in the current directory.

> You can also use an existing private key. When prompted for identity choice, select the import option and provide your key. It will be encrypted and stored in the Vault — the plaintext is not retained.

To reset the Vault and generate a fresh identity (for testing):

```bash
just reset-vault
```

---

## Step 3 — Review the Config

Open the generated `config.toml`. The critical fields for a first start:

```toml
[identity]
chain_id = 1                            # Target blockchain (1 = Ethereum mainnet)
rpc_url = "https://your-rpc-endpoint"  # JSON-RPC endpoint for on-chain reads
registry_address = "0x..."             # TruthOracleV3 contract address

[server]
rest_port = 3000    # REST API
admin_port = 3001   # Admin API (localhost-only, requires token)
admin_token = "..."  # Generated by 'init' — do not change to a placeholder

[p2p]
enabled = false     # Start in Sovereign mode. Set to true for Mesh Network.

[workers]
spawn_internal = true               # Let the gateway manage the Sidecar process
logic_host_cwd = "ts/sidecar"      # Sidecar working directory
logic_host_command = "pnpm run dev" # Sidecar start command
```

See the full [Configuration Reference](/gateway/configuration) for every available field.

**Security check:** The gateway refuses to start if:
- `admin_token` is empty or matches a known placeholder value.
- The private key matches the well-known Hardhat/Anvil test key (`0xac0974...`).

Both of these are hard errors, not warnings.

---

## Step 4 — Install at Least One Plugin

The gateway has no built-in data sources. Data comes entirely from installed plugins. Install a plugin before starting the node:

```bash
# Build and install a local plugin
just install-plugin ./path/to/my-plugin my-plugin-id

# Or use the CLI directly
./target/debug/hot-core --config config.toml plugin install ./path/to/my-plugin my-plugin-id --local
```

If the plugin requires an API key, store it in the Vault:

```bash
./target/debug/hot-core --config config.toml secrets set MY_PLUGIN_API_KEY <your-key>
```

The key will be injected into the plugin adapter at request time. It is never logged.

---

## Step 5 — Start the Gateway

```bash
# Using just (recommended for development)
just start

# Or directly
export GATEWAY_MASTER_PASSWORD=<your-password>
export RUST_LOG=info
./target/debug/hot-core --config config.toml start
```

Successful startup produces structured JSON logs. Key lines to look for:

```
{"level":"INFO","message":"Identity Bootstrapped","address":"0xYourNodeAddress"}
{"level":"INFO","message":"LogicRegistry fully initialized"}
{"level":"INFO","message":"Plugin preloaded successfully","id":"my-plugin-id"}
{"level":"INFO","message":"Node active","rest_port":3000,"admin_port":3001}
```

If you see `Plugin failed to preload`, the plugin has a configuration or dependency issue. The gateway continues starting — other plugins are unaffected.

---

## Step 6 — Verify the Node

```bash
# Basic health check
curl http://localhost:3000/health

# Discover installed capabilities
curl http://localhost:3000/discovery

# Run a diagnostic
./target/debug/hot-core --config config.toml doctor

# View node status (address, mode, plugin count)
./target/debug/hot-core --config config.toml status
```

Make a test fetch to confirm the plugin is working:

```bash
curl -X POST http://localhost:3000/fetch \
  -H "Content-Type: application/json" \
  -d '{"category": "crypto", "method": "spotPrice", "params": {"symbol": "BTCUSDT"}}'
```

A successful response includes `truthPoint`, `signature`, and `participatingSources`.

## Enabling Mesh Network Mode

To join or form a Mesh Network, update `config.toml`:

```toml
[p2p]
enabled = true

[workers]
enable_tss = true   # Enable BLS threshold signing
```

Restart the node. It will begin discovering peers via libp2p Kademlia DHT and participating in GossipSub consensus rounds.

> TSS is disabled by default. When `enable_tss = false`, attestations are signed with the node's local Ethereum key only — no multi-party quorum. A warning is printed at startup. For production Mesh deployments, always set `enable_tss = true`.

## Production Considerations

| Check | Action |
| :--- | :--- |
| Run as a non-root system user | Create a dedicated `taas` service user. |
| Use a process supervisor | `systemd` or `supervisord` to handle restarts. |
| Provide the Master Password securely | Use `--master-password-file /path/to/secret` and restrict file permissions to `0400`. Do not set `GATEWAY_MASTER_PASSWORD` in shell history. |
| Enable structured JSON logging | Default when `LOG_PRETTY` is unset. Forward logs to your observability stack. |
| Restrict the Admin API | `admin_port` binds to `127.0.0.1` by default. Do not expose it publicly. |
| Use a hardware-backed RPC endpoint | The `rpc_url` should be a reliable, authenticated endpoint. |
